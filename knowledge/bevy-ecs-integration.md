# Bevy ECS Integration for State Management

## Bevy Integration (`tut_11_bevy`)

The `examples/11_bevy/` app renders litui multi-page content inside a full Bevy application using `bevy_egui 0.40`. Compatibility: egui 0.34, bevy 0.19, wgpu 27.0.

### Architecture

```
tut_11_bevy (Bevy binary)
├── define_markdown_app! { ... }  ← markdown content files
├── PageRes / AppStateRes newtypes wrapping generated Page / AppState (bevy Resources)
├── render_nav system (EguiContexts, ResMut<PageRes>) — top panel navigation
└── render_page system (EguiContexts, Res<PageRes>, ResMut<AppStateRes>) — page dispatch
```

### Key findings

- litui macro-generated code (`render_*()` functions) works **unchanged** inside `bevy_egui`'s egui context
- No changes needed to the proc-macro, helpers, or content files
- First compile: ~51s (bevy + all deps). Incremental: ~1s
- `EguiPrimaryContextPass` schedule label for UI systems; `.chain()` for ordering nav before page
- `egui` crate must be an explicit dependency (macro-generated code references `egui::` directly)

### Running

```sh
cargo run -p tut_11_bevy   # Bevy version
cargo run -p tut_08_multi_page  # eframe version (same concepts)
```

## ECS Primer (Bevy 0.19)

- **World**: container for all ECS data
- **Resource** (`#[derive(Resource)]`): singleton state (one instance per type)
- **System**: plain `fn` with typed parameters (`Res<T>`, `ResMut<T>`)
- **Schedule**: ordered collection of systems

> **Bevy 0.19 requires Rust ≥ 1.95.** The litui workspace MSRV is pinned to 1.95.

## Using generated `Page` / `AppState` as Bevy Resources (bevy 0.19+)

`Page` and `AppState` are generated by `define_markdown_app!`. The litui macro stays
**bevy-agnostic** — it emits plain `Page`/`AppState` types and never references the `Resource`
trait. Wrapping them for Bevy happens in user/app code.

### What changed in bevy 0.19

Bevy 0.19 redefined the `Resource` trait with a `Component` supertrait:

```rust
pub trait Resource: Component {}
```

The old pattern — a bare `impl Resource for AppState {}` on a macro-generated type — **no longer
compiles**, because `Page`/`AppState` are not `Component`s and the empty impl doesn't make them
one. There is no longer a way to opt a foreign-defined plain struct into `Resource` with a single
empty impl.

### Canonical fix: newtype that derives `Resource`

Wrap the generated types in local newtypes and derive `Resource` on the wrapper. `#[derive(Resource)]`
alone is sufficient — the derive also satisfies the `Component` supertrait:

```rust
#[derive(Resource, Default)]
struct PageRes(Page);

#[derive(Resource, Default)]
struct AppStateRes(AppState);
```

Access the inner generated value through `.0`:

```rust
fn render_page(
    mut contexts: EguiContexts,
    page: Res<PageRes>,
    mut state: ResMut<AppStateRes>,
) {
    // read the current page
    match page.0 {
        Page::Home => { /* ... */ }
        // ...
    }
    // read/write AppState fields through .0
    let v = state.0.demo_slot;
    state.0.demo_slot = v + 1;
}
```

This is the pattern used by `examples/11_bevy/` and `examples/13_custom/`. It keeps the macro
bevy-agnostic (no Bevy dependency leaks into the generated code) while giving Bevy a type it owns
and can derive `Resource`/`Component` on.

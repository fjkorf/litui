# Bevy ECS Integration for State Management

## Bevy Integration (`tut_11_bevy`)

The `examples/11_bevy/` app renders litui multi-page content inside a full Bevy application using `bevy_egui 0.40`. Compatibility: egui 0.34, bevy 0.19.

### Architecture

```
tut_11_bevy (Bevy binary)
‚îú‚îÄ‚îÄ define_litui_app! { ... }  ‚Üê markdown content files
‚îú‚îÄ‚îÄ Page and AppState as Bevy Resources
‚îú‚îÄ‚îÄ render_nav system (EguiContexts, ResMut<Page>) ‚Äî top panel navigation
‚îî‚îÄ‚îÄ render_page system (EguiContexts, Res<Page>, ResMut<AppState>) ‚Äî page dispatch
```

### Key findings

- litui macro-generated code (`render_*()` functions) works **unchanged** inside `bevy_egui`'s egui context
- No changes needed to the proc-macro, helpers, or content files
- First compile: ~51s (bevy + all deps). Incremental: ~1s
- `EguiPrimaryContextPass` schedule label for UI systems; `.chain()` for ordering nav before page
- `egui` crate must be an explicit dependency (macro-generated code references `egui::` directly)

### Running

```sh
cargo run -p tut_11_bevy   # Bevy version
cargo run -p tut_08_multi_page  # eframe version (same concepts)
```

## ECS Primer (Bevy 0.19)

- **World**: container for all ECS data
- **Resource** (`#[derive(Resource)]`): singleton state (one instance per type)
- **System**: plain `fn` with typed parameters (`Res<T>`, `ResMut<T>`)
- **Schedule**: ordered collection of systems

## Using generated `Page` / `AppState` as Bevy resources (newtype pattern)

`Page` and `AppState` are generated by `define_litui_app!`. In **Bevy 0.19** the
`Resource` trait now requires `Component` as a supertrait (`Resource: Component`),
so the old `impl Resource for AppState {}` no longer works ‚Äî you cannot blanket-
impl `Resource` on a foreign-shaped generated type without also satisfying
`Component`.

The canonical pattern is to wrap each generated type in a newtype that **derives**
`Resource` (the derive also satisfies the `Component` supertrait) and access the
inner value through `.0`:

```rust
#[derive(Resource, Default)]
struct PageRes(Page);
#[derive(Resource, Default)]
struct AppStateRes(AppState);

// register
app.init_resource::<PageRes>()
   .init_resource::<AppStateRes>();

// systems access the generated type via `.0`
fn render_page(current: Res<PageRes>, mut state: ResMut<AppStateRes>, /* ... */) {
    match current.0 {
        Page::Mixed => render_mixed(ui, &mut state.0),
        // ...
    }
}
```

See `examples/13_custom/src/main.rs` for a complete working example
(`PageRes`/`AppStateRes` newtypes wired through `init_resource`, nav, and page
dispatch systems).
